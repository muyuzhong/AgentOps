# Phase 4.5 LLM Intent Judge Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fill the single LLM seam that Phase 4 left open. Behind the existing `IntentJudge` protocol, add an LLM-backed judge that decides, per deterministic scope finding, whether the gap falls *within the task's intent* (`within_intent`) or is real *drift* (`drift`) — turning the Phase 4 placeholder `needs_review` into an evidence-backed verdict. The deterministic default path stays byte-for-byte unchanged: no API key, no network, no new runtime dependency. The LLM judge is strictly opt-in and degrades to the deterministic verdict on any failure.

**Basis:** This is the follow-up the Phase 4 plan and roadmap point to, shaped by the spike answers in `docs/superpowers/findings/2026-06-03-scope-drift-spike.md`:

- detection is deterministic, the **verdict is not** → introduce the LLM only at the intent verdict, never to re-derive file sets;
- the model should receive *the declaration + the deterministic findings* and return a **per-finding verdict** (`drift` vs incidental-but-justified) — exactly the call the spike showed deterministic rules cannot make (`tests/test_auth.py` is within intent, `src/billing.py` is suspicious, yet both surface as identical `undeclared_change`);
- the seam already exists and is shaped correctly: `IntentJudge` is an injectable `Protocol`, `IntentVerdict` already carries `verdict ∈ {within_intent, drift, needs_review}` and `source ∈ {deterministic, llm}`, and `run_eval(..., intent_judge=...)` already accepts an injected judge. **This phase fills the seam; it does not reshape it.**

**Decisions locked for this slice** (from planning):

- **provider-agnostic** — define an in-house `LLMClient` protocol; the Anthropic adapter is the first concrete implementation, not the interface;
- **model tier deferred** — do not hard-code a model tier (haiku vs sonnet); the model id is supplied via CLI/env, and the right default is decided later from real eval data;
- **scope = the seam only** — no Phase 5 memory, no new evaluation dimensions, no verdict-driven scoring.

**Architecture:** Phase 4.5 inserts one provider-agnostic call inside the already-built `judge_intent` step. Nothing else in the `run_eval` pipeline changes.

```text
Phase 4 default (unchanged, no dependency):
  reconcile_scope → judge_intent[DeterministicIntentJudge] → build_eval_result → write
                    (intent_alignment → needs_review, source=deterministic)

Phase 4.5 opt-in:
  reconcile_scope → judge_intent[LLMIntentJudge]
                      ├─ build prompt: task goal/intent + agent's `changes` + each deterministic
                      │                drift finding (code + paths) + the changed-path set
                      ├─ LLMClient.complete(...)            # provider-agnostic; Anthropic adapter
                      ├─ parse + validate JSON → IntentVerdict(verdict ∈ {within_intent, drift}, source="llm")
                      └─ on ANY failure (no SDK / no key / no model / network / unparsable / invalid)
                         → degrade to DeterministicIntentJudge → needs_review, source="deterministic"
                  → build_eval_result → write
                  (score still comes from deterministic evaluate_scope; the verdict does NOT move it)
```

Core principle, unchanged: **the workflow controls the process; the LLM only enriches the intent verdict.** The verdict becomes richer (per-finding `within_intent`/`drift` instead of one `needs_review`), but the deterministic scope-discipline score remains fully reproducible and model-independent.

**Tech Stack:** Python 3.11+, standard library (`dataclasses`, `json`, `os`, `typing.Protocol`), `pytest`. Reuse `TaskReport`, `ScopeDriftReport`, `ScopeDriftFinding`, `IntentVerdict`, `IntentJudge`, `DeterministicIntentJudge`, `run_eval`, `EvalReportWriter`. **New optional dependency:** the Anthropic SDK, declared as an `llm` extra — never in the core runtime dependency list. All tests use a stub `LLMClient`; no test touches the network.

## Scope Guard

Implement only the LLM-backed intent judge behind the existing seam, its provider-agnostic client, and the opt-in CLI wiring.

Do not add in this phase:

- **verdict-driven scoring** — the score still comes from the deterministic `evaluate_scope`; a `drift` verdict does not deduct and a `within_intent` verdict does not refund. (Letting a non-deterministic verdict move a reproducible score is deferred until Phase 5 can calibrate it against accumulated data.)
- **changes to the seam shape** — do not modify `core/eval.py` (`IntentVerdict`, verdict/source constants) and do not change the `IntentJudge.judge(task_report, report)` signature. Intent judging uses path-level context (goal + `changes` + finding paths); feeding raw diff hunks would require widening the interface and is a later enhancement, not this slice.
- **a default model tier baked into code** — the model id is supplied by the caller (CLI flag / env var); no `haiku`/`sonnet` literal in library code.
- **a second provider implementation** — define the provider-agnostic `LLMClient` protocol and exactly one adapter (Anthropic). Other providers are future adapters.
- the Phase 5 memory store or trend reports (the LLM verdicts simply enrich the existing `eval-history.jsonl` lines);
- evaluation dimensions beyond scope/boundary; multi-task or multi-session batch evaluation; a watcher / real-time monitoring; automatic edits to application source or to `CLAUDE.md` / `AGENTS.md`.

The default `agentops eval` (no `--intent-judge llm`) MUST remain identical to Phase 4: deterministic, read-only with respect to the target repo, no LLM/network/API-key dependency. `agentops eval` writes only under the chosen `--output` directory.

## User-Visible Result

After Phase 4.5:

```powershell
# Unchanged default — deterministic, offline, no key:
agentops eval --repo <repo-path>

# Opt-in LLM intent judge:
agentops eval --repo <repo-path> --intent-judge llm --intent-model <model-id>
```

- `--intent-judge {deterministic,llm}` defaults to `deterministic` (the Phase 4 behavior);
- `--intent-model <id>` selects the model when `--intent-judge llm`; the API key is read from the environment (e.g. `ANTHROPIC_API_KEY`);
- with `--intent-judge llm`, each deterministic drift finding receives a per-finding verdict (`within_intent` / `drift`, `source="llm"`) with a one-line rationale, written into `agentops-report.md`, `agentops-score.json`, and the appended `eval-history.jsonl` line;
- if the SDK is missing, the key is unset, the model id is absent, the call fails, or the response is unparsable/invalid, the judge **degrades to the deterministic `needs_review` verdict**, the eval still succeeds (exit 0), and a single concise line is written to stderr explaining the fallback;
- the report groups findings by verdict so a reviewer sees at a glance which undeclared changes are *confirmed drift* versus *likely justified*.

## Contracts

### Provider-agnostic LLM client (new seam)

```python
# agentops/llm/client.py
@dataclass(frozen=True)
class LLMRequest:
    """一次裁决调用的 provider 无关请求。"""
    system: str
    user: str
    max_tokens: int = 1024
    temperature: float = 0.0          # 裁决要尽量可复现，温度取 0


@dataclass(frozen=True)
class LLMResponse:
    """provider 无关的文本响应；结构化解析交给调用方（judge）。"""
    text: str


class LLMError(RuntimeError):
    """统一封装所有 LLM 调用失败：缺 SDK、缺 key、网络、超时、provider 报错。"""


@runtime_checkable
class LLMClient(Protocol):
    """文本进、文本出的最薄接缝；具体 provider 由适配器实现。"""
    def complete(self, request: LLMRequest) -> LLMResponse: ...
```

The protocol is deliberately text-in/text-out so any provider can satisfy it. Structured output is the judge's responsibility (strict-JSON prompt + parse + validate + degrade), which keeps the seam provider-agnostic and makes an unparsable response a safe fallback rather than a crash.

```python
    def complete(self, request: LLMRequest) -> LLMResponse:
        # 延迟 import anthropic：缺包 → LLMError；缺 key → LLMError；网络/SDK 错误 → LLMError。
        # system 作为可缓存块（prompt caching，cache_control），user 作为单条消息。
        ...
```

- `model` is required — no tier is hard-coded in library code (decision: model tier deferred).
- The `anthropic` import is lazy and lives only here, so importing `agentops` never requires the SDK.
- The system block carries the stable judging instructions and is marked cacheable (prompt caching), since it repeats across evals.

### LLM-backed intent judge (fills the seam)

```python
# agentops/judges/llm_intent.py
class LLMIntentJudge:
    """用可注入 LLMClient 对确定性漂移 finding 逐条裁决意图归属；任何失败降级到确定性。

    实现现有 IntentJudge 协议，不改变其签名。降级时内部委托
    DeterministicIntentJudge，保证受限环境下行为与默认路径完全一致。
    """
    def __init__(self, client: LLMClient, *, fallback: IntentJudge | None = None) -> None: ...

    def judge(
        self, task_report: TaskReport, report: ScopeDriftReport
    ) -> tuple[IntentVerdict, ...]:
        ...
```

Behavior:

1. **Select reviewable findings** — the deterministic drift findings worth an intent verdict: `undeclared_change` and `cross_module_breadth` (per-path / per-breadth). If `report.findings` has no `intent_alignment` trigger (`llm_needed=True`), there is nothing to judge → return `()` (same as the deterministic judge on a clean report).
2. **Build the prompt** — system: the judge's role and the hard rule *"the file sets are already determined; judge only whether each listed gap fits the task intent — do not re-derive file sets"*; user: `task_report.goal` (intent), `task_report.changes` (what the agent said it did), the full `changed_paths` set as context, and each reviewable finding (`code` + `evidence` paths). Request a strict JSON array of `{finding_code, evidence, verdict, rationale}`.
3. **Parse + validate** — JSON parses, every object has a `verdict ∈ {within_intent, drift}`, and the set of returned `(finding_code, evidence)` covers the reviewable findings. On success, map to `IntentVerdict(..., source="llm")`.
4. **Degrade on any failure** — `LLMError`, JSON parse error, schema mismatch, or incomplete coverage → delegate to the `fallback` (default `DeterministicIntentJudge`), returning its `needs_review` / `source="deterministic"` verdict(s). The judge never raises out of `judge()` for a model-side problem.

This reuses `IntentVerdict` verbatim (`finding_code`, `evidence`, `verdict`, `rationale`, `source`) — **no change to `core/eval.py`.** The Phase 4 default emitted exactly one `needs_review`; the LLM path may emit several per-finding verdicts. That is the intended enrichment.

### Scoring boundary (unchanged)

`build_eval_result` keeps calling `evaluate_scope(scope)` for the score and the scope `Finding`/`Recommendation` set. `intent_verdicts` is the LLM's enrichment and is recorded alongside, exactly as in Phase 4 — it does not feed the score in this slice.

### CLI wiring (opt-in, fail-safe)

- `--intent-judge {deterministic,llm}` (default `deterministic`); `--intent-model <id>`.
- A patchable factory builds the judge: `_resolve_intent_judge(args) -> IntentJudge`. For `llm`, it constructs an `AnthropicClient(model=..., api_key=os.environ.get(...))` wrapped in `LLMIntentJudge`. Tests monkeypatch this factory (or a `_build_llm_client` helper) to inject a stub client, so CLI tests never touch the network.
- If `--intent-judge llm` is requested but the model id is missing or the client cannot be constructed, the CLI falls back to `DeterministicIntentJudge`, prints one concise stderr line (e.g. `agentops: intent judge fell back to deterministic (no --intent-model)`), and the eval proceeds with exit 0.

## Exit Criteria

Phase 4.5 is complete when:

- `agentops eval --repo <p> --intent-judge llm --intent-model <id>` returns per-finding `within_intent` / `drift` verdicts with `source="llm"`, written into the report, score JSON, and the appended `eval-history.jsonl` line;
- any failure (no SDK / no key / no model / network error / unparsable / invalid response) degrades to the deterministic `needs_review` (`source="deterministic"`), the eval still exits 0, and a single concise stderr notice explains the fallback;
- the default `agentops eval` (no `--intent-judge llm`) is identical to Phase 4 — deterministic, offline, no API key, no new runtime dependency (the Anthropic SDK lives only in the `llm` extra and is imported lazily);
- the `LLMClient` protocol is provider-agnostic, `AnthropicClient` is the one adapter, and `import agentops` works without the SDK installed;
- `core/eval.py` (`IntentVerdict`, verdict/source constants) and the `IntentJudge.judge(...)` signature are unchanged;
- the deterministic scope-discipline score is unaffected by the verdict (a degrading run scores the same as the default run);
- every test injects a stub client and none touches the network; `python -m pytest -v` passes;
- `scan` / `init` / `check-session-log` behavior is unchanged.

The next step after Phase 4.5 is to plan **Phase 5 repository memory** on top of the now verdict-enriched `eval-history.jsonl` (history evals, failure modes, rules, skill candidates), and to revisit whether accumulated `drift` verdicts justify letting the intent verdict calibrate the score.
